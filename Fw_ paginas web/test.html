Ruta de Aprendizaje — 5 Etapas
01
Fundamentos de Testing
Semana 1 · 5 días · Base sólida desde cero
▾

DÍA 01

¿Por qué testear? Costo de bugs. Anatomía de un test: Arrange → Act → Assert. Tu primer TestCase.

DÍA 02

Assertions esenciales: assertEqual, assertTrue, assertRaises. Correr tests con manage.py test.

DÍA 03

Setup y teardown. Fixtures en JSON. Aislamiento entre tests. Por qué cada test debe ser independiente.

DÍA 04

Testeando modelos Django: creación, validaciones, métodos custom, __str__, Meta options.

DÍA 05

Revisión integrada. Refactorización de tests. Lectura del output de errores. Preguntas abiertas.

Reto Etapa 1: Crea un modelo Producto con nombre, precio y stock. Escribe mínimo 8 tests que cubran: creación válida, validación de precio negativo, método esta_disponible(), y representación __str__. Todos deben pasar con manage.py test.

02
Vistas, URLs y Autenticación
Semana 2 · 5 días · Testear lo que el usuario ve
▾

DÍA 01

El Client de Django. GET requests. Verificar status codes y redirecciones.

DÍA 02

POST requests. Testear formularios Django. Manejo del token CSRF en tests.

DÍA 03

Login/logout en tests. force_login(). Probar vistas con permisos y restricciones.

DÍA 04

Testear templates: assertTemplateUsed. Verificar contexto de vistas. Mensajes flash.

DÍA 05

Revisión. Organización de tests por apps. Naming conventions profesionales.

Reto Etapa 2: Para tu app de Productos, crea tests de: listado (solo usuarios autenticados), creación (POST válido e inválido), detalle (404 si no existe), y eliminación (solo el dueño puede borrar). Mínimo 12 tests.

03
Técnicas Intermedias
Semana 3 · 5 días · Herramientas que usan los pros
▾

DÍA 01

factory_boy: genera datos de prueba realistas sin fixtures manuales. Factories con relaciones.

DÍA 02

unittest.mock: simular APIs externas, emails, servicios de pago. patch y MagicMock.

DÍA 03

Testear signals de Django. Tests de middleware custom. Interceptar emails con mail.outbox.

DÍA 04

Django REST Framework: testear endpoints de API. APIClient. Autenticación JWT en tests.

DÍA 05

Revisión. Refactorizar tests anteriores con factories. Comparar antes y después.

Reto Etapa 3: Refactoriza todos tus tests anteriores usando factory_boy. Agrega tests para: un endpoint de API que consulta precios externos (mockeado), y una signal que envía email al crear un Producto (verificar con mail.outbox).

04
Calidad, Cobertura y Performance
Semana 4 · 5 días · Tests rápidos y medibles
▾

DÍA 01

pytest-django: fixtures de pytest, markers, plugins. Por qué es mejor que TestCase puro.

DÍA 02

coverage.py: generar reportes HTML. Interpretar resultados. Estrategia para subir cobertura.

DÍA 03

Tests parametrizados con @pytest.mark.parametrize. Eliminar duplicación de tests.

DÍA 04

Optimizar velocidad: setUpTestData, tests paralelos, base de datos en memoria, caché.

DÍA 05

Revisión. Configurar pytest.ini y .coveragerc para un proyecto real.

Reto Etapa 4: Migra tu suite completa a pytest. Logra 90%+ de cobertura en tu app de Productos. Reduce el tiempo de ejecución a menos de 3 segundos usando las técnicas aprendidas. Documenta los cambios de velocidad antes/después.

05
Nivel Profesional — TDD y CI/CD
Semana 5 · 5 días · Como lo hacen en producción
▾

DÍA 01

TDD en la práctica: Red → Green → Refactor. Desarrollar una feature completa guiada por tests.

DÍA 02

GitHub Actions: workflow para correr tests en cada PR. Badges de cobertura. Bloquear merge si fallan.

DÍA 03

Tests de integración reales. Introducción a Playwright para tests E2E de flujos críticos.

DÍA 04

Organización de suites grandes. Testing en equipos. Code review de tests. Estrategias reales.

DÍA 05

🏆 Proyecto Final: app completa con suite profesional, CI configurado y cobertura documentada.

🏆 Proyecto Final: Construye usando TDD una app de Carrito de Compras con: modelos, API REST, autenticación, signal de confirmación por email, 95%+ cobertura, pipeline de CI en GitHub Actions que bloquea el merge si los tests fallan. Este proyecto va a tu portafolio.
